# Agentic AI 기반 신약개발 PoC 구현 보고서

## 1. 보고 개요

본 문서는 AWS 기술 블로그에서 소개된 SK바이오팜형 Agentic AI 신약개발 사례를 참고하여, 동일한 개념의 PoC(Proof of Concept)를 당사 환경에서 어떻게 구현할 것인지에 대한 실행 방안을 정리한 보고서입니다.

이번 PoC의 목적은 후보물질 평가와 Go/No-Go 의사결정 과정에서 필요한 정보 수집, 분자 특성 예측, 문헌 및 임상 근거 분석, 전문가 관점 종합 판단을 하나의 워크플로로 자동화하는 것입니다.

---

## 2. 추진 배경

신약개발의 Lead Optimization 단계에서는 후보물질의 구조, 효능, 독성, 약동학, 임상 가능성, 규제 전략을 동시에 검토해야 합니다. 그러나 실제 연구 환경에서는 PubChem, ChEMBL, PubMed, ClinicalTrials.gov, openFDA 등 여러 소스를 연구자가 직접 탐색해야 하며, 이 과정은 시간이 오래 걸리고 결과의 일관성도 확보하기 어렵습니다.

특히 다음과 같은 문제가 반복적으로 발생합니다.

- 후보물질 하나를 평가하기 위해 여러 데이터베이스를 반복 검색해야 함
- 화학, 임상약리, 규제 관점의 정보가 분절되어 종합 판단이 어려움
- 연구자 개인의 경험에 따라 분석 품질이 크게 달라짐
- Go/No-Go 판단을 위한 근거 정리가 늦어져 전체 개발 일정이 지연됨

따라서 이번 PoC는 Agentic AI 기반 멀티 에이전트 구조를 활용하여, 연구자가 SMILES와 질문만 입력하면 필요한 분석과 리포트가 자동으로 생성되는 시스템을 구현하는 것을 목표로 합니다.

---

## 3. 구축 목표

이번 PoC의 목표는 다음과 같습니다.

- 연구자가 입력한 SMILES를 기반으로 분자 특성을 자동 예측
- 공개 데이터 소스에서 화합물, 문헌, 임상, 규제 정보를 병렬 수집
- 화학, 임상약리, 규제 전략 전문가 역할의 AI 에이전트가 병렬 분석 수행
- 최종적으로 경영진 또는 연구책임자 관점의 Go/No-Go 리포트 생성
- 자연어 질문에 따라 필요한 전문가만 선택하는 스마트 컨설팅 기능 제공

이번 단계는 PoC이므로 실제 운영 시스템이 아니라, 핵심 사용자 흐름과 기술 실현 가능성을 입증하는 수준으로 구현합니다.

## 5. 제안 아키텍처

본 PoC는 `로컬 개발 + AWS 연동` 구조로 설계합니다.

### 5.1 구성 요소

- **UI 계층**
  - Streamlit 웹앱
  - 연구자가 SMILES, 타겟, 질문을 입력
  - 예측 결과, 에이전트 분석 결과, 최종 판단 리포트 출력

- **API 계층**
  - FastAPI 서버
  - 분석 요청 접수, 검증, 워크플로 실행, 응답 조립 담당

- **분자 처리 계층**
  - RDKit 기반 SMILES 검증 및 canonicalization
  - 분자 구조 SVG 생성

- **예측 계층**
  - Amazon SageMaker에 배포한 TxGemma 엔드포인트 호출
  - 핵심 10개 속성에 대한 예측 수행

- **데이터 수집 계층**
  - PubChem, ChEMBL, PubMed, ClinicalTrials.gov, openFDA API를 병렬 호출
  - 수집된 데이터를 에이전트별 사용 목적에 맞게 정규화

- **에이전트 계층**
  - Walter White: 구조/물성/SAR 분석
  - Dr. Gregory House: PK/PD, DDI, 독성, 안전성 분석
  - Harvey Specter: 임상/규제/승인 전략 분석
  - CEO/Router: 종합 판단 또는 질문별 에이전트 선택

- **캐시 계층**
  - 동일 SMILES 반복 요청 시 예측값과 외부 근거 재사용
  - 로컬 디스크 캐시 사용

### 5.2 처리 흐름

#### A. 종합 분석 흐름

1. 사용자가 SMILES와 타겟 정보를 입력
2. 시스템이 RDKit으로 유효성을 검증하고 canonical SMILES 생성
3. TxGemma로 핵심 10개 속성을 예측
4. 외부 데이터 소스에서 관련 근거를 병렬 수집
5. Walter, House, Harvey 세 에이전트를 동시에 실행
6. CEO 집계 에이전트가 결과를 종합하여 최종 리포트 생성
7. UI에 Go/Conditional Go/No Go 판단과 근거 표시

#### B. 스마트 컨설팅 흐름

1. 사용자가 자연어 질문 입력
2. Router가 질문 의도를 파악하여 필요한 전문가 선택
3. 선택된 에이전트만 병렬 분석 수행
4. CEO 또는 통합 응답 계층이 최종 답변 생성
5. 선택된 전문가와 근거 출처를 함께 제시

## 6. 핵심 기능 정의

### 6.1 분자 특성 예측

v1에서는 TxGemma를 활용해 Go/No-Go 판단에 직접 영향이 큰 핵심 10개 속성을 우선 구현합니다.

- Solubility
- LogP/LogD
- Permeability
- Plasma Protein Binding
- Clearance
- Half-life
- CYP Inhibition
- hERG Risk
- Hepatotoxicity
- BBB Penetration

이 10개는 향후 22개까지 확장 가능하도록 `속성 레지스트리` 구조로 설계합니다.

### 6.2 멀티 에이전트 종합 분석

세 전문가 에이전트는 동일한 입력을 공유하되, 각자 다른 데이터와 관점을 중심으로 분석합니다.

- Walter는 유사 화합물, SAR, 구조 최적화, 물성 개선 제안을 담당
- House는 예측값과 문헌을 바탕으로 PK/PD, DDI, 독성 리스크를 판단
- Harvey는 임상 개발 가능성, 승인 허들, 경쟁 및 전략적 우선순위를 평가

각 에이전트는 자유형 문장이 아니라 구조화된 JSON 결과를 반환하고, 최종 응답 계층에서 이를 사람이 읽기 쉬운 한국어 문서로 변환합니다.

### 6.3 스마트 컨설팅

질문 예시는 다음과 같습니다.

- "이 화합물의 심장독성 위험은?"
- "FDA 승인 가능성이 있나?"
- "구조 개선 방향은 무엇인가?"

Router는 질문을 해석해 관련 전문가를 선택합니다. 예를 들어 심장독성 질문은 House와 Walter를, 승인 가능성 질문은 Harvey와 House를 우선 선택하도록 설계합니다. 라우팅 결과가 불확실하면 안전하게 세 전문가를 모두 호출합니다.

## 7. 시스템 인터페이스

### 7.1 종합 분석 API

`POST /api/reports/executive`

입력

- `smiles`
- `target`
- `compound_name` 선택값

출력

- `canonical_smiles`
- `molecule_svg`
- `predictions`
- `evidence_bundle`
- `agent_findings`
- `executive_decision`
- `citations`

### 7.2 스마트 컨설팅 API

`POST /api/reports/consult`

입력

- `smiles`
- `target`
- `question`
- `compound_name` 선택값

출력

- `selected_agents`
- `routing_reason`
- `predictions`
- `agent_findings`
- `consulting_answer`
- `citations`

## 12. 리스크 및 대응 방안

### 리스크 1. 외부 API 응답 편차

공개 데이터 소스는 응답 형식과 검색 품질이 일정하지 않을 수 있습니다.

대응 방안

- 각 소스별 정규화 계층 분리
- 실패 시 전체 중단이 아니라 부분 결과 허용
- 출처별 수집 성공 여부를 리포트에 명시

### 리스크 2. 모델 응답의 비정형성

LLM 기반 에이전트는 출력 형식이 흔들릴 수 있습니다.

대응 방안

- strict JSON 출력 강제
- Pydantic 검증
- 파싱 실패 시 재시도 또는 전체 전문가 호출

### 리스크 3. 예측 신뢰도 검증 한계

PoC 단계에서는 예측값 자체를 규제 수준으로 검증하기 어렵습니다.

대응 방안

- 연구 보조용으로만 사용 범위 명시
- 최종 리포트에 참고용임을 표시
- 후속 단계에서 내부 실험 데이터와 비교 검증 추진

### 리스크 4. 비용 및 지연 시간

Bedrock, SageMaker, 다중 외부 API 호출로 인해 응답 시간이 길어질 수 있습니다.

대응 방안

- 캐시 적용
- 질의 유형별 전문가 선택 최적화
- 속성 수를 10개로 제한해 초기 응답 시간 관리
